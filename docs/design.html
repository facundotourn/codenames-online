Codenames Online

Segunda versión: multiplayer real con backend, salas y estado sincronizado en tiempo real.

Este documento describe la arquitectura y el diseño de la v2. Está pensado para revisarse antes de escribir código: cuando lo apruebes (o lo ajustemos), pasamos a scaffoldear el repo.

1. Visión y objetivo

Hoy la v1 es un SPA estático: el "multiplayer" es un truco determinístico — todos cargan la misma semilla (código de "Sala") y generan el mismo tablero, pero los reveals son locales y no se comparten. No hay turnos ni roles reales.

La v2 convierte eso en multiplayer de verdad: salas creadas en un servidor, con estado autoritativo y sincronizado en tiempo real entre todos los jugadores, e implementando las reglas completas de Codenames (spymaster, pista, turnos, adivinanzas).

4. Arquitectura

El navegador de cada participante abre un WebSocket contra la instancia de servidor de su sala (un Durable Object, identificado por el código de sala). Esa instancia tiene el estado autoritativo, valida cada acción y hace broadcast del nuevo estado — redactado según el rol de cada destinatario.

4.1 Clientes neutrales y modo "mesa compartida"

Además de los jugadores de equipo, una sala admite participantes neutrales (sin equipo). El caso que motiva esto: una juntada presencial donde se comparte la partida en un televisor y solo los espías juegan desde el celular.

  • Mesa compartida (tableBoard): la pantalla grande. No ve colores (es pública), pero puede revelar cartas en el turno de cualquiera de los dos equipos — los operativos se juntan alrededor y tocan ahí, sin necesidad de tener cada uno un dispositivo.
  • Espectador (spectator): mira la partida con la vista redactada, sin poder actuar.
Setup de juntada: el TV abre la sala como tableBoard y cada espía entra desde su celular como spymaster (rojo/azul). Los operativos no necesitan dispositivo: ven el tablero en la TV y tocan ahí en su turno.

La tableBoard funciona como un "operativo universal": a efectos de revelar (y terminar turno) actúa por el equipo que esté en turno en ese momento — pero nunca ve los colores ocultos, porque es una pantalla pública.

5. Hosting y estructura del repo

Un único repo. PartyKit guarda server y cliente juntos y comparten TypeScript directamente — no hace falta separar back y front en repos distintos.

# codenames-online/
party/
  server.ts      // clase Server: conexiones + máquina de estados + validación
  game.ts        // generación de tablero + lógica de reglas (pura)
  types.ts       // tipos compartidos cliente ↔ servidor
src/
  components/    // Board, Card (portados de la v1) + lobby, turno, pista…
  net/           // hook usePartySocket + capa de mensajes
  App.tsx
  data/words.ts  // lista de palabras (reusada de la v1)
partykit.json    // config de deploy
vite.config.ts
package.json
docs/design.html // este documento

Deploy: npx partykit deploy sube el server a los Durable Objects de Cloudflare y puede servir también el build de Vite. Un comando, una plataforma, free tier holgado.

6. Modelo de dominio

Tipos compartidos por cliente y servidor (party/types.ts):

type Team = 'red' | 'blue';
type CardColor = 'red' | 'blue' | 'neutral' | 'assassin';

// Roles de equipo (requieren team) + roles neutrales (sin team)
type Role = 'spymaster' | 'operative' | 'tableBoard' | 'spectator';
type Phase = 'lobby' | 'awaitingClue' | 'guessing' | 'finished';

interface Player {
  id: string;            // persistente, guardado en localStorage
  name: string;          // único en la sala; si se repite, se agrega " (n)" con n = 1..9
  role: Role;
  team: Team | null;     // spymaster/operative ⇒ Team; tableBoard/spectator ⇒ null
  connected: boolean;
  ready: boolean;        // listo en el lobby; cambiar de rol lo resetea a false
}

interface Card {
  id: string;
  word: string;
  color?: CardColor;     // verdad solo del server; se redacta para no-spymasters
  revealed: boolean;
}

interface Clue {
  word: string;
  count: number;          // N: cantidad de palabras conectadas
  team: Team;
  guessesUsed: number;    // hasta N + 1
}

interface GameState {
  phase: Phase;
  hostId: string;          // anfitrión; único que inicia. Si se va, migra al más antiguo conectado
  players: Record<string, Player>;
  board: Card[];
  startingTeam: Team;       // arranca con 9 cartas
  turn: Team;               // equipo en turno
  clue: Clue | null;        // pista vigente durante 'guessing'
  remaining: { red: number; blue: number };
  winner: Team | null;
}

7.1 Lobby: roles, "ready" y arranque

El arranque no es libre: lo controla el host y solo cuando todos están listos.

  • Host (anfitrión): el primer participante en entrar a la sala. Es el único que puede iniciar la partida. Si se desconecta, el rol migra automáticamente al participante conectado más antiguo (no hace falta que sea de un equipo: puede ser la propia TV).
  • Al entrar: cada uno elige equipo (rojo/azul) con rol spymaster u operativo, o se queda neutral (mesa compartida / espectador). Por defecto, al entrar se es espectador hasta elegir.
  • Cambiar de equipo/rol resetea tu estado a no listo (hay que volver a confirmar).
  • "Ready": cada jugador de equipo se marca listo cuando está. Los neutrales (mesa/espectador) no bloquean el arranque.
  • Roles bloqueados al iniciar: una vez empezada la partida no se puede cambiar de equipo/rol. Para reorganizar, el host usa "Cambiar equipos" (returnToLobby).

El botón "Iniciar partida" (visible solo para el host) se habilita cuando se cumplen las tres condiciones:

  1. Ambos equipos tienen ≥1 spymaster.
  2. Cada equipo puede adivinar: ≥1 operativo en ese equipo o una mesa compartida en la sala.
  3. Todos los jugadores de equipo (spymaster/operativo) están ready.
Decidido: los neutrales no se marcan ready ni bloquean el arranque — un espectador que entra tarde no debería frenar el inicio, y la TV suele estar lista antes que nadie.

8. Reglas del juego implementadas

  • Tablero: 25 cartas — 9 del equipo inicial, 8 del otro, 7 neutrales, 1 asesino.
  • Roles: cada equipo tiene al menos un spymaster (ve los colores); las adivinanzas las hacen los operativos del equipo o una mesa compartida neutral (ver §4.1).
  • Pista: el spymaster del turno envía una palabra + un número N. La palabra no puede estar en el tablero.
  • Adivinanzas: los operativos del turno (o la mesa) revelan cartas; tienen hasta N + 1 intentos.
  • Resolución por carta: color propio → siguen; neutral → fin de turno; rival → fin de turno y ayuda al rival; asesino → derrota instantánea.
  • Cortar antes: el equipo puede terminar su turno voluntariamente.
  • Victoria: el primer equipo en revelar todas sus cartas gana.

9. Anti-cheat: redacción por rol

El punto crítico del multiplayer real: no se puede confiar en el cliente. En la v1, el modo "Espía" es un toggle local — cualquiera lo prende y ve los colores. En la v2, los colores ocultos nunca viajan al cliente de quien no es spymaster.

El servidor arma una vista distinta por destinatario antes de cada envío:

  • Spymaster: recibe el board completo, con color en cada carta.
  • Operativo, mesa compartida y espectador: reciben el board redactado — el campo color se omite en toda carta no revelada; solo aparece cuando revealed === true. Esto es especialmente crítico en la mesa compartida, que es una pantalla pública mirada por todos.
function viewFor(state: GameState, role: Role): GameState {
  if (role === 'spymaster') return state;        // único rol que ve colores
  return {
    ...state,
    board: state.board.map(c =>
      c.revealed ? c : { ...c, color: undefined }   // se borra el color oculto
    ),
  };
}

MensajePayloadDestino

Con 25 cartas, mandar el GameState completo (redactado) en cada cambio es lo más simple y rinde de sobra. Nada de diffs ni eventos granulares por ahora.

11. Flujo de una ronda

  1. El spymaster rojo (turno) envía giveClue {"OCÉANO", 2}. El server valida y pasa a guessing.
  2. Broadcast del nuevo state: todos ven la pista. Tienen hasta 3 intentos los operativos rojos — o quien toque en la TV (mesa compartida).
  3. Se revela "BARCO" (tocando en la TV o en un celular). El server resuelve: es roja → remaining.red--, sigue el turno.
  4. Adivinan "AGUA" (también roja) → sigue. Luego arriesgan "NUBE" → es neutral: fin de turno.
  5. El server cambia turn = blue, vuelve a awaitingClue y hace broadcast. Ahora juega el spymaster azul.
  6. Cuando un equipo lleva su remaining a 0 (o alguien toca el asesino), phase = finished y se anuncia el ganador → 🎉 confeti (también en la TV).

12. Frontend

Se reusa de la v1

  • Board y Card (grilla 5×5, caras, flip 3D).
  • El flip dramático ("Suspenso final") y el confeti de victoria — ahora disparados por el estado sincronizado.
  • Tema oscuro, estilos, lista de palabras (data/words.ts).

Nuevo

  • Pantalla de entrada: crear / unirse a sala + nombre.
  • Lobby: elegir equipo y rol — incluyendo neutrales (mesa compartida o espectador); ver la lista de participantes con su estado ready; cada jugador se marca listo; el host ve el botón "Iniciar partida" (deshabilitado hasta cumplir la condición de §7.1).
  • HUD de partida: indicador de turno, pista vigente (palabra + número), intentos restantes, botón "terminar turno".
  • Panel del spymaster: input de pista + número (solo visible en su turno).
  • Pantalla de fin: el host elige entre "Nueva partida" (mismos equipos) o "Cambiar equipos" (volver al lobby).
  • Modo mesa / TV: layout de pantalla grande para la tableBoard — tablero protagonista, pista y turno bien visibles, tap para revelar; sin controles de rol propios.
  • Capa de red: hook usePartySocket que recibe state y envía acciones; reemplaza al useReducer local.

13. Sugerencia de pista (IA)

El backend desbloquea limpio el feature que veníamos charlando: el server tiene la ANTHROPIC_API_KEY en una variable de entorno y responde a requestClueSuggestion del spymaster del turno.

  • El server arma el prompt con el estado real del tablero (palabras de cada equipo, neutrales, asesino) — info que ya tiene de forma autoritativa.
  • Modelo claude-haiku-4-5 (rápido y barato; alcanza de sobra).
  • Restricciones en el prompt: una sola palabra, que no esté en el tablero ni sea derivada; el número de palabras que conecta; evitar a toda costa el asesino.
  • Rate-limit por sala para acotar costo, ya que la key vive segura en el server.
  • La respuesta se envía solo al spymaster que la pidió (clueSuggestion) — nunca a la TV ni a los operativos.

14. Persistencia y reconexión

  • En memoria mientras la sala está viva (lo natural de un Durable Object).
  • Snapshot en DO Storage tras cada cambio, para sobrevivir reinicios y permitir reconexión: si un participante recarga, su playerId (de localStorage) lo reconecta a su asiento — útil para la TV, que no querés que pierda su lugar.
  • Sala abandonada: si todos los participantes quedan desconectados, la sala (y la partida en curso) se descarta tras 5 minutos sin que nadie se reconecte.
